Data: 2026-04-18
Iteracao: API de interacoes (13.7)
Resumo: endpoints `POST` e `GET` em `/api/friends/{friend_id}/interactions` implementados. Ao registrar uma interacao, o servico atualiza `friend.last_contact_at` somente se a nova `occurred_at` for mais recente que o valor atual. 53 testes passando (27 dominio + 15 amigos + 11 interacoes).
Decisoes:
- rotas aninhadas ao amigo (`/api/friends/{id}/interactions`) tornam o escopo explicito e dispensam filtro por amigo no nivel global
- `last_contact_at` so avanca: interacao historica (occurred_at no passado) nao rebaixa o contato mais recente ja registrado — protege contra import de historico sobrescrever um ping recente
- `occurred_at` ausente no payload vira `datetime.now(ZoneInfo(settings.timezone))` no servico (registro rapido em um toque)
- datetimes naive vindos do SQLite sao tratados como timezone do app antes de comparar com `occurred_at` tz-aware, reusando o mesmo padrao de `friendship._aware`
- validacao de existencia do amigo usa `session.get(Friend, id)` antes de inserir/listar, devolvendo `FRIEND_NOT_FOUND` no formato padrao (mesmo path de erro do 13.6)
- listagem ordena por `occurred_at DESC, id DESC` — id desempata quando varias interacoes no mesmo timestamp (caso de import em lote)
- tipo invalido e nota > 2000 chars viram `VALIDATION_ERROR` automaticamente pelos schemas Pydantic existentes (nada a mudar em 13.4)
- deletar amigo ja apaga interacoes em cascata (FK com `ondelete=CASCADE` do 13.3) — coberto por teste

Iteracao: API CRUD de amigos (13.6)
Resumo: router `/api/friends` implementado com GET/POST/PATCH/DELETE e filtros opcionais (category, cadence, tag). Servico `friend_service` hidrata `FriendRead` com as metricas de `friendship`. Formato de erro padrao (secao 9.7 do PRD) implementado em `app/errors.py` com handlers globais. 42 testes passando (27 de dominio + 15 de integracao de API).
Decisoes:
- criado modulo `app/errors.py` com `AppError`, `NotFoundError`, `ConflictError`, `ExternalServiceError` e handlers que serializam no formato `{"error": {"code","message","details"}}` (inclui `RequestValidationError` -> code `VALIDATION_ERROR` com status 400 e `HTTPException` -> code generico)
- tags sao normalizadas (`strip().lower()`) e deduplicadas preservando ordem no `friend_service._unique_tags`; filtro por tag aceita qualquer caixa
- `selectinload(Friend.tags)` usado em todas as queries para evitar `MissingGreenlet` em contexto async
- apos `create_friend`/`update_friend`, em vez de `session.refresh(..., attribute_names=["tags"])` (que disparava lazy load e quebrava em async), re-seleciona o amigo com `get_friend(session, id)` para devolver o objeto ja hidratado
- `PATCH` usa `payload.model_dump(exclude_unset=True)` para atualizacao parcial real (nao zera campos nao enviados)
- listagem sempre ordenada por `name` para previsibilidade da UI
- dependencia `pydantic` promovida para `pydantic[email]` no `pyproject.toml` pois `EmailStr` exige `email-validator`
- fixtures de teste usam arquivo SQLite em `tmp_path` (nao `:memory:`) porque conexoes diferentes do pool precisam enxergar as mesmas tabelas
- `conftest.py` faz override de `get_db` com `async_sessionmaker(expire_on_commit=False)` e commita ao final de cada request para que leituras subsequentes no mesmo teste vejam os dados
